# 智能驾驶舱视觉审计与回归验证

更新时间：2026-05-30

## 目标

本文分为两部分：

1. 沉淀当前智能驾驶舱的视觉审计与设计改进基线
2. 记录本轮实际执行过的回归测试结果，并给出下一步具体改进意见

## 当前视觉问题

### 1. 整体入口层面

- 入口页当前“可用”，但还不像一个高价值的智能驾驶舱产品，更像是功能列表页。
- 左侧导航、空状态、驾驶舱卡片、创建入口之间缺少一个统一的“产品主叙事”。
- 用户进入系统后，第一眼看不到三个核心判断：
  - 我现在能做什么
  - 当前系统是否健康
  - 哪些驾驶舱最值得我先打开

### 2. 驾驶舱列表层面

- 驾驶舱卡片的信息组织比较平均，缺少主次关系。
- 当前卡片里名称、描述、组件数、更新时间、智能体信息都在展示，但“最关键的业务摘要”不够突出。
- 预览区域还比较弱，只能看到零散 widget 片段，不能快速感知这个驾驶舱是“战略型、财务型、运营型，还是实时监控型”。

### 3. 创建链路层面

- 创建弹窗已经比之前统一，但视觉上仍像一个“配置表单”，不像“智能生成工作流”。
- 模板与自由创建虽然逻辑在收敛，但感知上还是两套体验：
  - 模板是标签选择
  - 自由创建是文本输入
- 当前流程提示是文字化阶段条，信息正确，但缺乏“系统正在理解、规划、生成、初始化”的沉浸感与信任感。

### 4. 单个驾驶舱详情页层面

- 详情页整体已经具备可视化驾驶舱雏形，但主要问题是“层次还不够稳”。
- Widget 壳层、标题栏、内容密度、比例关系，在数据复杂时容易显得挤或散。
- 目前很多 widget 还是“组件在显示数据”，而不是“数据在驱动布局”。
- 页面缺少一个更强的总览带：
  - 驾驶舱是什么
  - 当前由谁驱动
  - 当前数据状态如何
  - 是否真实数据 / 演示数据 / 初始化失败

### 5. 视觉语言层面

- 当前浅色主题的红色主色要求是正确的，但使用方式还有提升空间。
- 红色主要用在按钮是对的，但整体页面的中性色层级还不够高级：
  - 背景层和卡片层接近
  - 悬浮层、选中层、编辑层之间差异还不够清晰
- 部分区域仍偏“开发态界面”，不够像正式产品。

影响：

- 但会影响首屏加载、页面切换与移动端体验

建议：

- 对管理页、设置面板、详情页大模块做延迟加载
- 对图表与少用组件做拆包

## 分层优化建议

## 一、驾驶舱入口页

建议把入口页从“列表页”升级成“工作台首页”。

建议新增 4 个区块：

- 顶部欢迎区
  - 展示一句明确价值主张，例如“面向业务与智能体协同的驾驶舱工作台”。
  - 右上角保留主按钮“新建驾驶舱”，但左侧增加一句系统状态摘要。
- 最近使用区
  - 不是纯列表，而是突出 3-5 个最近/高频/告警中的驾驶舱。
- 模板与场景区
  - 展示 CFO、CEO、运营、供应链、研发、人力等场景入口，用卡片而不是小标签。
- 系统状态区
  - 展示连接状态、模板数量、活跃驾驶舱、异常驾驶舱数。

设计原则：

- 入口页不要太多红色。
- 用友红只用于主 CTA、关键强调、激活态。
- 背景建议改成更有层次的暖灰白 + 极轻的纹理/渐变，而不是纯平涂。

## 二、左侧导航与驾驶舱卡片

### 导航建议

- 导航顶部不要只写“驾驶舱”，建议加入更完整品牌识别，如“智能驾驶舱工作台”。
- 驾驶舱列表建议分组：
  - 最近使用
  - 高价值业务驾驶舱
  - 其他驾驶舱
- 当前选中态可以更稳：
  - 左侧细红线保留
  - 再叠加轻底色和更清晰的标题字重差异

### 卡片建议

- 卡片结构改成“标题区 + 核心摘要区 + 状态脚注区”。
- 每张卡片建议增加 1 个主摘要指标，而不是只展示名称和描述。
- 智能体状态、真实数据状态、更新时间建议合并成一行微型状态条。

推荐的卡片信息层级：

1. 驾驶舱名称
2. 一句话业务价值
3. 1-2 个高价值摘要指标
4. 状态脚注：真实数据/演示数据、组件数、更新时间、智能体健康度

## 三、创建弹窗与创建链路

当前建议把创建弹窗升级成“双栏智能创建面板”。

左栏：

- 创建方式
- 模板选择
- 驾驶舱名称
- 用户目标输入

右栏：

- 智能生成说明
- 预计生成内容
- 参考模板
- 数据来源预期
- 风险提示

建议把“自由创建”和“模板创建”的视觉结构也完全统一：

- 不是“选不选模板”的两套界面
- 而是“输入目标 -> 系统参考模板与知识库 -> 生成驾驶舱”的同一条链路

创建过程浮层建议从 toast 升级成更有产品感的“右下角任务卡”：

- 顶部显示当前阶段
- 中部显示逐步生成说明
- 底部显示已完成步骤
- 完成后可以“一键打开驾驶舱”

## 四、单个驾驶舱详情页

建议把单个驾驶舱页面分成 4 层：

### 1. 驾驶舱总览条

放在页面最上方，包含：

- 驾驶舱名称和一句话说明
- 主智能体 / 协作智能体
- 数据模式：真实数据 / 演示数据 / 混合
- 最近刷新时间
- 当前过滤器与视图状态

### 2. 核心指标带

不是所有 widget 都直接平铺，建议优先抽出顶部“关键指标带”：

- 3 到 5 个主 KPI
- 更大字号
- 更稳定的比例
- 明确趋势和异常语义

### 3. 主内容画布

主画布分三类区域：

- 趋势分析区
- 结构分析区
- 明细与事件区

这样比“所有 widget 同级平铺”更有叙事顺序。

### 4. 智能协作与操作区

把当前底部聊天/命令区做成更像“驾驶舱协作台”：

- 谁在执行
- 执行了什么
- 当前建议动作
- 会话历史

## 五、Widget 系统的视觉方向

### 当前问题

- Widget 标题栏风格已经形成，但壳层的差异化还不够。
- 不同类型 widget 的内部布局密度差异太大。
- 指标卡过于单值中心，复杂业务信息表达不够。
- `universal` 目前更像“文本容器”，还不像真正的“复杂信息兜底容器”。

### 建议方向

#### 1. 指标卡升级

把 `metric` 升级成支持三种形态：

- 单主指标
- 主指标 + 2~4 个次级指标
- 主指标 + 对比对象 + 说明标签

#### 2. 通用兜底组件升级

建议新增更强的 `adaptive` / `agent-panel` 类组件：

- 作为智能体生成内容的统一容器
- 内部按 sections/cards/blocks 渲染
- 可以展示摘要、列表、状态、表格、指标组合、洞察文本
- 极端情况下，可以独立完成特殊数据的承载

#### 3. 尺寸体系重算

建议默认尺寸改成更接近真实内容密度：

- `metric`：`4x2`
- `progress`：`4x2`
- `status` / `list` / `alert`：`4-5x3`
- `chart` / `table` / `timeline` / `kanban` / `adaptive`：`6x4`
- `report` / `html`：`8x4`

这样会比现在大量 `3x2` 的卡片更稳。

## 六、视觉语言建议

### 保持不变

- 浅色主题保留“用友红”为主强调色
- 红色继续只用于按钮、激活态、重要提示、关键操作

### 建议优化

- 页面背景加入更轻的层次感：
  - 暖灰白底
  - 很浅的方向性渐变
  - 几乎不可感知的颗粒/网格纹理
- 卡片边框做得更细、更轻
- 悬浮态更依赖阴影和亮度变化，而不是重边框
- 编辑态、选中态、错误态、真实数据态，要建立清晰的颜色语义

### 字体与数字建议

- 中文正文字体继续以系统中文字体为基础，保证稳定性
- 标题和数字建议引入更有辨识度的数字表达方式
- KPI 数字、百分比、金额应统一 tabular 数字风格

## 七、最值得优先落地的 3 个视觉改造

### P1

- 继续收窄首页欢迎区高度，让欢迎语更像“背景中的品牌层”，而不是一个独立大卡片。
- 将首页主结构稳定为“顶部品牌头 + 欢迎语 + 驾驶舱实例画布 + 可展开快速创建”，避免容器层级继续增加。
- 快速创建区域默认建议保持展开，但需要保证展开/收起的高度变化更平滑，避免页面跳动感。

### P2

- 创建弹窗继续保持“简化版统一方案”，不要再回到复杂双栏。
- 但要补 3 个细节：
  - 给模式切换区增加更明确的激活态对比，减少用户对“当前到底是自由创建还是模板创建”的疑惑
  - 模板模式下，名称自动填充后应明确提示“可继续修改”
  - 模板模式和自由模式的提交按钮文案已经区分，但建议再增加一行轻提示，说明当前将走哪条创建链路

### P3

- 单个驾驶舱详情页继续简化头部，保持现在的紧凑高度是对的。
- 下一步重点不是加更多信息，而是优化 widget 之间的比例关系：
  - 顶部关键指标更稳
  - 图表区更完整
  - 明细区不过分挤压
- 协作智能体展示要继续回到“默认只突出主智能体”的逻辑，多智能体只在真实存在时展开展示。

## 具体改进意见

以下建议按优先级排序，优先考虑“高价值、低风险、能直接提升稳定性和体验”的项。

### A. 高优先级：一周内应处理

#### 1. 清理创建链路中的 UI 状态歧义

建议：

- 给创建弹窗增加一个只读状态标签，例如：
  - `当前模式：自由创建`
  - `当前模式：模板创建 · CEO 战略总览`
- 当从首页预制入口进入时，模板模式和模板选择要绝对同步，不允许只填充 prompt 而不切换模式

原因：

- 这条链路用户之前多次反馈过问题
- 即使当前本地回归通过，也仍属于高敏感路径

#### 2. 修复并清理 `meta-agent` 关键词映射噪音

建议：

- 清理 `server/src/services/meta-agent.ts` 中重复的 `指标卡`
- 按“组件名 -> 渲染类型 -> 兜底类型”重整映射表

原因：

- 这类小问题很容易继续积累，最终影响 AI 选组件稳定性

#### 3. 给创建失败和初始化失败补更明确的错误分层

建议：

- 区分：
  - 创建请求失败
  - 模板创建成功但数据初始化失败
  - 真实数据获取失败但演示数据可兜底
- 在右下角统一创建提示卡里明确表达

原因：

- 这直接关系到用户是否能理解“到底是没创建成功，还是数据初始化失败”

### B. 中优先级：两周内建议处理

#### 4. 首页驾驶舱卡片补“更强的业务摘要”

建议：

- 每张卡片保留 1 个主摘要指标
- 再补 1 个场景标签，如：
  - `战略总览`
  - `财务经营`
  - `运营效率`
  - `天气服务`

原因：

- 现在卡片已可用，但“识别这是什么驾驶舱”的速度还不够快

#### 5. 统一首页预制入口与实例卡片的操作反馈

建议：

- 出厂预制卡片和实例卡片继续保持同一视觉体系
- 但 hover 和点击反馈要区分：
  - 预制卡片强调“快速创建”
  - 实例卡片强调“进入驾驶舱”

原因：

- 这样更符合你的产品设想，也能减少“看起来一样但行为不同”的认知落差

#### 6. 进一步优化 widget 默认尺寸与排布规则

建议：

- 继续以 `4x2 / 4x3 / 6x4 / 8x4` 为主尺寸体系
- 对 `metric`、`status`、`list` 三类高频组件优先做密度和比例校准

原因：

- 当前整体已能用，但很多内容仍是“能展示”，还没达到“好展示”

### C. 中长期：下一阶段演进建议

#### 7. 给首页增加“系统健康与连接状态”的弱表达

建议：

- 不要做成大监控面板
- 只在头部或欢迎区右侧用轻量摘要表达：
  - 活跃驾驶舱数
  - 连接状态
  - 初始化异常数

原因：

- 这能让首页更像“企业级工作台”，但不至于重新变复杂

#### 8. 为组件系统建立正式预览与回归集

建议：

- 后续围绕组件管理，建立组件级 demo 数据预览
- 针对 `metric / chart / adaptive / universal` 固化一组标准回归样本

原因：

- 这能把“视觉优化”和“组件稳定性”统一到一套验证体系里

## 结论

当前系统在本轮回归后，已经达到“继续演进但不影响基本可用”的状态。

最值得优先处理的不是大规模重做，而是：

1. 继续压实创建链路的状态一致性
2. 清理 AI 选组件映射和错误提示中的小噪音
3. 进一步提升首页卡片摘要与 widget 比例的专业度

如果下一步继续推进，我建议把工作拆成两条并行线：

- 一条是“创建链路与组件稳定性”
- 一条是“首页卡片与 widget 表达质量”

这样可以同时守住可用性和持续提升产品完成度。

- 入口页升级为“工作台首页”
- 创建弹窗升级为“智能创建面板”
- 单驾驶舱顶部增加“总览条 + 数据状态条”

### P2

- widget 默认尺寸重算
- metric 多指标化
- adaptive 通用容器落地

### P3

- 导航和卡片分组体系
- 聊天协作区产品化
- 全局动效与主题精修

## 当前回归风险

### 1. 产品标题仍是原型名

- 首页 HTML `<title>` 仍然是 `Agentic Chat UI Prototype`
- 这会拉低整体产品完成度，建议尽快品牌化

### 2. Widget 尺寸体系仍不统一

- 运行中的模板和历史驾驶舱大量仍使用旧的 `3x2` 指标卡逻辑
- 视觉上会继续出现“碎、小、密”的问题

### 3. 通用组件承载能力仍偏弱

- 模板中依然存在用 `universal` 承载复杂数据的情况
- 这会导致信息表达力和视觉稳定性不足

### 4. 存在失败态但视觉信任表达不足

- 某些驾驶舱中能看到 `__initStatus: failed` 的初始化失败数据
- 但当前页面未形成足够强的“失败原因 / fallback 状态 / 建议动作”视觉反馈

## 建议的下一步实施顺序

1. 先做入口页与创建弹窗的视觉升级，不碰复杂 widget 逻辑。
2. 再做单驾驶舱顶部总览条和状态条。
3. 再推进 widget 尺寸体系与 `metric` / `adaptive` 升级。
4. 最后统一细节动效、阴影、背景层次和空状态语言。
