# YonCockpit 历史会话说明与重建上下文

更新时间：2026-05-29

本文不是代码说明，而是“这个项目为什么会长成现在这样”的上下文沉淀。目标是让后续重新从零创建工程时，能够把正确的问题重新问出来，把正确的边界重新立起来。

## 1. 本文来源

本文综合了四类信息：

1. 当前仓库代码结构
2. 仓库内现有设计文档
3. Git 历史
4. 本轮会话中的明确用户意图

## 2. Git 历史能反映出的演进

当前可见的提交历史很短，但足以说明项目的三次关键阶段：

1. `eda5e63 Initial commit`
2. `1f5f4a0 Add subtle elevation and adjust theme tokens`
3. `fa4f922 feat: expand cockpit orchestration and dashboard editing`

这说明项目演化不是“先做完整平台再拆子模块”，而是：

- 先有一个可运行的前端/后端原型
- 再做 UI 风格和交互表层细化
- 再快速扩展到连接、模板、编排、画布编辑和 Meta-Agent

## 3. 从仓库文档能看出的产品背景

仓库中的设计文档表明，这个项目不是凭空出现的，而是来自更大的 YonClaw 设想。

### 3.1 上位愿景

上位愿景来自：

- [design_specification.md](/Users/jiang/Documents/GitHub/cockpit-human/design_specification.md:1)

其核心概念是：

- YonClaw 是企业超级智能体入口
- 用户通过自然语言触发智能体理解、编排、界面生成
- 驾驶舱是 Agentic GUI 的一种核心呈现形态

### 3.2 驾驶舱子问题收敛

与当前仓库更接近的设计来自：

- [design/multi-agent-orchestration.md](/Users/jiang/Documents/GitHub/cockpit-human/design/multi-agent-orchestration.md:1)

它把问题收敛成：

- 驾驶舱如何绑定多个智能体
- 外部平台 Agent 与内置驾驶舱智能体如何切换主导权
- 协作状态如何在 UI 中可见

### 3.3 当前仓库的真实定位

因此，`cockpit-human` 当前真实定位可以定义为：

> YonClaw 大平台中的“智能驾驶舱子应用”，它独立承载驾驶舱创建、展示、对话、模板化和多智能体协作呈现。

## 4. 本轮会话中明确下来的用户意图

本轮会话里，用户连续表达了几类很关键的诉求。

### 4.1 诉求一：先理解现状

用户先要求“整体阅读当前文件夹工程”，说明当前阶段不是盲目开发，而是先统一认知。

这意味着未来重建时，也应该先做“结构梳理”和“边界定义”，而不是直接开写。

### 4.2 诉求二：保留历史地上传仓库

用户随后要求把当前版本上传到 GitHub，且“历史版本需要保留”。

这说明：

- 项目已经从一次性原型转向持续迭代资产
- 历史演进本身对项目有价值
- 工程治理虽然还轻，但用户已经在意版本延续性

### 4.3 诉求三：公开仓库按纯源码整理

在上传过程中，本轮会话明确沉淀了一个重要原则：

- 公开仓库应优先保留源码、文档、测试和必要的静态资产
- 运行态数据、PID、测试残留、连接配置不应直接进入仓库

这会影响未来重建时的默认目录策略：

- demo 数据和系统模板可以入库
- 本地运行快照和敏感连接配置不应该入库

### 4.4 诉求四：要求形成“可重建”的知识沉淀

当前用户并不只要一个能跑的项目，而是要：

- 详细的产品需求说明
- 应用架构说明
- 技术架构说明
- 一份能支撑“后续从零再造”的历史会话说明

这意味着未来工程不应再依赖“只有当前开发者脑中知道”的隐性背景。

## 5. 当前工程的核心设计决策

这些决策是在代码和会话中都被反复确认过的，未来重建时应默认继承。

### 5.1 必须是 mock-first

原因：

- 用户需要先看见成品和流程
- 外部平台并不总是可用
- 模板、画布和驾驶舱体验不应该依赖真实平台才可开发

因此重建时第一阶段必须有 mock adapter。

### 5.2 驾驶舱是产品中心，不是附属页面

当前应用不是“对话为主，驾驶舱只是一个结果页”，也不是“BI 为主，对话只是边栏”。

正确定位是：

- 驾驶舱是主工作区
- 对话能力是驾驶舱内部的一等交互方式

### 5.3 模板与自然语言创建必须并存

原因：

- 模板创建适合标准场景和快速起步
- 自然语言创建适合探索性需求和演示智能能力

两者缺一不可。

### 5.4 多智能体协作状态必须可见

不是只在后端算一个状态，而是要在前端呈现：

- 当前主控是谁
- 当前是 platform-led 还是 cockpit-led
- 健康状态如何
- 哪些 agent 正在参与

### 5.5 连接管理是产品功能，而不是仅开发配置

原因：

- 当前系统希望支持多个外部平台
- 用户需要在 UI 中切换和测试连接
- 连接、能力和编排状态之间是业务相关的，不只是 dev-only 配置

### 5.6 驾驶舱自身需要向外暴露为 Meta-Agent

这说明系统不是只做“消费型前端”，而是也要成为更大智能体生态中的“供给方”。

## 7. 从零重建时的推荐顺序

为了避免重建时再次走弯路，建议按下面的顺序，而不是按“先把所有页面画出来”的方式。

### 第一步：先确定产品骨架

先固定四个核心对象：

- Workspace
- Widget
- Template
- Connection

再固定三个核心状态：

- Agent mode
- Orchestration state
- Widget data source

### 第二步：只做 mock 可运行闭环

先实现：

- 工作区列表
- 工作区详情
- mock adapter
- 静态 widget 渲染
- 基础聊天

目标不是完整，而是得到第一个可演示闭环。

### 第三步：加入模板系统

实现：

- 内置模板
- 从模板创建 workspace
- 模板管理基础页

这一步很关键，因为它把“展示型原型”变成“可扩展产品”。

### 第四步：加入自然语言创建

实现：

- `CockpitAgent`
- 意图识别
- 任务规划
- workspace 自动创建

### 第五步：加入连接与编排

实现：

- ConnectionManager
- Connector 抽象
- AgentRouter
- Orchestrator

### 第六步：加入实时反馈

实现：

- EventBus
- WebSocket 事件流
- 初始化状态反馈
- 编排状态变化反馈

### 第七步：加入 Widget 数据管道

实现：

- `static`
- `skill`
- `query`
- `event`

### 第八步：最后补平台化能力

实现：

- Meta-Agent
- 更细的工具定义
- 对外调用协议

## 8. 从零重建时不要丢掉的约束

1. 驾驶舱必须可以在没有真实外部平台的情况下跑起来。
2. Workspace 必须是最核心的业务对象。
3. 模板和自然语言创建必须并存。
4. 多智能体编排状态必须是显式模型，不要藏在 UI 逻辑里。
5. 事件流必须存在，因为初始化和编排变化都不是同步瞬时动作。
6. 连接管理必须纳入 UI，不要退化为纯 `.env` 配置。

## 9. 给未来重新创建工程的提示词草案

如果未来需要让新的工程师或新的 agent 从零重建，可以直接使用下面的约束说明作为起始输入：

> 请创建一个名为 YonCockpit 的前后端同仓应用。  
> 它是 YonClaw 超级智能体平台中的“智能驾驶舱子应用”，核心对象是 Workspace。  
> 用户必须能浏览多个驾驶舱、通过模板或自然语言创建驾驶舱、在详情页以网格形式查看和编辑 Widget、并通过底部聊天区与当前驾驶舱智能体对话。  
> 系统必须支持 mock-first 运行，不依赖真实平台即可演示。  
> 同时需要有连接管理、模板管理、WebSocket 事件流、SSE 对话、基础多智能体编排状态，以及将驾驶舱暴露为 Meta-Agent 的后端能力。  
> 前端技术栈使用 React + Vite + TypeScript，后端使用 Express + TypeScript。  
> 请优先保证领域模型清晰、模块边界清晰、可扩展到连接器与编排器，而不是只做静态页面原型。

## 10. 当前这一轮会话的最终沉淀

本轮会话最终明确了三件事：

1. 当前仓库已经值得被当成持续演化工程，而不是一次性演示稿。
2. 需要把“代码现状”与“产品/架构意图”分开沉淀，避免未来只剩代码没有上下文。
3. 后续如果从零重建，目标不是一比一抄当前实现，而是保留其正确的产品边界、模块划分和演化顺序。
