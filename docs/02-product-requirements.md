# YonCockpit 产品需求说明

更新时间：2026-05-29

本文以当前 `cockpit-human` 代码实现为基准，结合仓库内既有设计文档与本轮会话上下文，抽取出“从零重建时仍应保留”的产品需求。

## 1. 产品定位

YonCockpit 不是传统静态 BI Dashboard，而是一个面向企业场景的“智能驾驶舱应用”：

- 用户可以创建多个业务驾驶舱
- 每个驾驶舱可绑定外部智能体或由内置智能体独立运行
- 用户既可以像看 dashboard 一样浏览数据，也可以像用 copilot 一样直接对话和下达命令
- 驾驶舱既是展示界面，也是智能体协作和业务操作的工作台

更准确地说，YonCockpit 是从更大 YonClaw 超级智能体平台中收敛出来的一个独立应用子集，重点解决：

- 多智能体协作结果如何可视化呈现
- 驾驶舱如何通过模板和自然语言快速生成
- 外部平台 Agent、LLM 与本地 UI 如何打通

## 2. 目标用户

### 2.1 业务用户

典型角色：

- 销售负责人
- HR/行政运营
- 财务审批人员
- 运维/客服负责人
- 营销分析人员

核心诉求：

- 快速创建一个可用的业务驾驶舱
- 在一个界面里看到指标、图表、表格、告警和洞察
- 直接用自然语言让系统解释数据、补充组件、调整布局

### 2.2 平台管理员 / 集成人员

典型角色：

- AI 平台管理员
- 解决方案实施顾问
- 开发/集成工程师

核心诉求：

- 配置 OpenClaw / YonClaw / Hermes / 通用 LLM 连接
- 管理驾驶舱模板
- 观察事件流、调试协作状态
- 将当前驾驶舱作为 Meta-Agent 暴露给外部平台调用

## 3. 产品目标

### 3.1 业务目标

- 让“创建一个能用的驾驶舱”从手工开发缩短到分钟级
- 让用户不离开驾驶舱即可完成“看数 + 追问 + 操作”
- 让多智能体协作过程具备可见的状态和最基本的可解释性

### 3.2 体验目标

- 首次进入即可看到示例驾驶舱或空欢迎页
- 新建驾驶舱既支持模板，也支持自然语言
- 驾驶舱详情页既支持浏览，也支持低门槛编辑
- 智能体协作状态、事件和初始化过程需要有反馈

## 4. 核心需求

### 4.1 驾驶舱列表与切换

系统必须支持：

- 展示全部驾驶舱列表
- 至少支持 `cards`、`sidebar`、`tabs` 三种工作区组织方式
- 记住用户最近选择的驾驶舱和布局模式
- 删除驾驶舱后自动清理无效选中状态与 tab

验收标准：

- 用户刷新页面后仍能恢复最近选择
- 驾驶舱为空时显示明确的欢迎/引导状态

### 4.2 驾驶舱创建

系统必须支持两种创建入口：

- 基于模板创建
- 基于自然语言命令创建

模板创建需求：

- 允许从系统模板和自定义模板中选取
- 可选传入新名称
- 可选传入初始化提示词
- 创建后可后台触发 LLM 初始化数据

自然语言创建需求：

- 通过 `CockpitAgent` 识别意图
- 规划出驾驶舱规格
- 创建成功后自动跳转/打开新驾驶舱

约束：

- 驾驶舱总量上限当前为 30

### 4.3 驾驶舱详情页

详情页必须同时承担“展示层”和“操作层”两种职责。

展示能力：

- 显示名称、描述、图标、主题色
- 展示当前协作模式和健康状态
- 展示关联智能体及其主从关系
- 以网格方式渲染组件

操作能力：

- 修改名称和描述
- 拖拽/调整组件布局
- 删除已有组件
- 从组件库新增组件
- 打开组件详情抽屉

### 4.4 Widget 体系

系统当前应支持如下 Widget 类型：

- `metric`
- `chart`
- `table`
- `list`
- `kanban`
- `timeline`
- `report`
- `universal`
- `progress`
- `status`
- `html`
- `gauge`
- `funnel`
- `radar`
- `heatmap`
- `bullet`
- `alert`
- `map`

每个 Widget 需要支持：

- 标题
- 布局坐标与尺寸
- 静态数据
- 可选动态数据源
- 可选详情配置
- 可选跳转配置

### 4.5 多智能体协作与调度状态

每个驾驶舱需要具备：

- 主智能体
- 协作智能体列表
- 协作模式 `single / multi-coordinator / multi-parallel / llm-only`
- 运行态编排状态 `platform-led / cockpit-led / llm-direct`
- 健康度 `healthy / degraded / unavailable`

体验要求：

- 用户能看到当前是谁在主控
- 用户能看到驾驶舱是外部平台主导还是内置智能体接管
- 当外部 Agent 不可用时，系统应可降级到 cockpit-led 或 llm-direct

### 4.6 驾驶舱对话能力

每个驾驶舱都必须支持聊天入口。

聊天行为要求：

- 优先走 `CockpitAgent` 智能对话链路
- 如果失败，回退到 workspace 原有 adapter chat
- 流式显示回复内容
- 本地保存最近对话历史

聊天内容目标：

- 回答当前驾驶舱指标/组件相关问题
- 解释当前协作状态
- 接受新增组件、删除组件、分析趋势等命令

### 4.7 连接管理

系统必须支持平台连接管理：

- 新增连接
- 编辑连接
- 删除连接
- 测试连接
- 手动连接
- 手动断开

当前支持的连接类型：

- `generic-llm`
- `yonclaw`
- `openclaw`
- `hermes`

每条连接必须具备：

- 地址
- 协议
- 能力声明
- 优先级
- 启用状态
- 健康状态

### 4.8 模板管理

系统需要有管理员侧模板管理页。

能力要求：

- 查看所有模板
- 新建模板
- 编辑模板
- 复制内置模板为自定义模板
- 删除模板
- 直接从模板创建驾驶舱

模板属性至少包括：

- 名称
- 领域
- 关键词
- 图标
- 颜色
- 关联智能体
- 组件定义
- 初始化提示词
- 是否启用 demo fallback

### 4.9 Widget 动态数据能力

Widget 需要支持四类数据源：

- `static`
- `skill`
- `query`
- `event`

行为要求：

- 静态数据可直接渲染
- `skill` 和 `query` 可由后端路由到外部能力
- `event` 类型需要能承载实时事件驱动场景
- 数据失败时可根据 workspace 策略决定回退到 demo 数据或空数据

### 4.10 事件流与实时反馈

系统必须通过 WebSocket 暴露事件流，至少覆盖：

- `workspace.created`
- `workspace.initializing`
- `workspace.initialized`
- `workspace.init_failed`
- `workspace.updated`
- `orchestration.changed`
- 连接/平台事件

前端需基于事件流做到：

- 创建/初始化的即时反馈
- 设置面板中可查看事件列表
- 驾驶舱状态变化时更新提示

### 4.11 Meta-Agent 对外暴露

驾驶舱应用应具备“被外部平台当作一个智能体调用”的能力。

当前最少应暴露的工具包括：

- `cockpit_plan`
- `cockpit_create`
- `cockpit_execute`
- `cockpit_update`
- `cockpit_query`
- `cockpit_list`
- `cockpit_schedule`
- `agent_list`
- `agent_invoke`
- `connection_list`
- `connection_test`

## 5. 非功能需求

### 5.1 可演示性

- 默认可在无外部平台的情况下运行
- mock 适配器必须能支撑本地演示

### 5.2 可重建性

- 前后端边界清晰
- 核心领域模型应明确
- 模板、工作区、连接需要可持久化
- 关键文档应足以支持从零重建

### 5.3 可扩展性

- 允许新增 Widget 类型
- 允许新增连接器
- 允许新增模板
- 允许未来接入更强的编排与鉴权机制

## 6. 当前范围边界

### 已在当前版本范围内

- 驾驶舱管理
- 模板管理
- 连接管理
- 对话
- 编排状态展示
- Widget 编辑与数据刷新
- Meta-Agent 暴露

### 明确尚未闭环

- 真正安全的管理员权限体系
- 完整的事件驱动 Widget 自动更新
- 可靠的生产级存储与并发控制
- 更细粒度的多用户/多租户能力
- 深链接、分享、权限分层

## 7. 从零重建时必须优先保留的产品不变量

1. 这是“智能驾驶舱应用”，不是纯聊天工具，也不是纯 BI 页面。
2. 驾驶舱必须兼具“可视化展示”和“自然语言操作”两种入口。
3. 多智能体协作状态必须是产品一等公民，而不是隐藏实现细节。
4. 模板创建、自然语言创建、连接配置是三个同等重要的入口。
5. 必须支持无外部平台的 mock 演示模式，否则开发和演示门槛过高。
