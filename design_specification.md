# YonClaw 超级智能体平台 — 产品原型设计说明

**版本：** v3.0  
**日期：** 2026-05-23  
**产品定位：** 企业超级智能体入口与调度中心  

---

## 一、产品定位与核心理念

### 1.1 YonClaw 是什么

YonClaw 不是传统意义上的企业 SaaS 产品，而是一个**企业超级智能体入口（Enterprise Super-Agent Hub）**和**服务调度中心（Service Orchestration Center）**。它是用户与企业所有业务能力之间的唯一交互界面——用户通过自然语言与 YonClaw 主智能体对话，由主智能体理解意图、调度子智能体、聚合数据、动态生成界面，最终完成业务目标。

这一设计哲学与 Salesforce Headless 360 的"Build once, render everywhere"理念一脉相承，也与 SAP Joule 的"woven-in（织入式）"架构方向一致。不同之处在于，YonClaw 更进一步：不仅是 Agent 调用 Headless API，而是 Agent 直接生成用户界面本身——这是真正的 **Agentic GUI** 形态。

### 1.2 从 GUI-Centric 到 Agentic-Centric 的范式转移

传统企业软件的交互范式经历了四个阶段的演进：

| 阶段 | 代表产品 | 用户行为 | 系统响应 |
|------|----------|----------|----------|
| **工具时代** | SAP ECC / Oracle EBS | 点击菜单 → 找到功能 → 填写表单 | 固定页面，被动等待 |
| **搜索时代** | 企业搜索 + 快捷入口 | 搜索关键词 → 进入功能 | 搜索结果列表 |
| **Copilot 时代** | Microsoft Copilot / GitHub Copilot Chat | 侧边栏提问 → 获得建议 | 文本回复 + 简单操作 |
| **Agentic 时代** | **YonClaw** | 自然语言表达意图 | **动态理解 → 调度Agent → 生成界面 → 完成闭环** |

YonClaw 的设计目标是在第四阶段建立一套完整的**Agent-Native 交互体系**：用户不再适应软件的界面，软件通过智能体重塑自身以适应用户的表达。

### 1.3 平台架构总览

YonClaw 平台由六大核心模块构成，形成从"意图理解"到"界面生成"到"业务闭环"的完整链路：

```
┌─────────────────────────────────────────────────────────────┐
│                    用户交互层（呈现层）                        │
│  ┌─────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐       │
│  │ 对话界面 │ │智能驾驶舱│ │ 智能团队 │ │ 友智库   │       │
│  │  (Chat) │ │(Workspace)│ │ (Agents) │ │(Knowledge)│      │
│  └─────────┘ └──────────┘ └──────────┘ └──────────┘       │
├─────────────────────────────────────────────────────────────┤
│                    YonClaw 主智能体（调度中心）               │
│         意图理解 · 任务分解 · Agent编排 · UI生成             │
├─────────────────────────────────────────────────────────────┤
│                    子智能体层（业务能力层）                    │
│  销售助手 │ HR助手 │ 财务管家 │ 营销智脑 │ IT运维 │ 法务 │ 客服 │
├─────────────────────────────────────────────────────────────┤
│                    工作流编排层                              │
│         触发器 → 条件分支 → Agent调用 → 审批 → 通知 → 结束   │
├─────────────────────────────────────────────────────────────┤
│                    企业数据与工具层（MCP Server）              │
│         CRM │ ERP │ HCM │ 财务系统 │ 业务中台 │ 知识库      │
└─────────────────────────────────────────────────────────────┘
```

## 二、核心模块设计详解

### 2.1 对话界面 — 一切交互的起点

对话界面是 YonClaw 的主入口，承载着"意图唤醒"的核心职责。它遵循 Ant Design X 提出的 **RICH 设计范式**（Role · Intention · Conversation · Hybrid UI），将用户与 AI 的交互抽象为四个递进阶段：

**唤醒（Awaken）阶段** —— 解决"YonClaw 能做什么"的认知问题。首页欢迎消息附带能力速览卡片，展示销售数据查询、任务创建、审批处理、排行榜查看、工作流管理五大能力域，降低用户准入门槛。

**表达（Express）阶段** —— 降低用户表达意图的成本。输入框下方集成快捷指令按钮（技能创建、定时任务、智能审批、搜索技能、全网检索），用户可一键触发高频场景，无需记忆复杂的自然语言表达方式。

**确认（Confirm）阶段** —— 缓解等待焦虑。Agent 回复采用**流式打字效果**逐字呈现，模拟真实推理过程。复杂任务展示思考过程和进度指示，让用户始终感知系统正在工作。

**反馈（Feedback）阶段** —— 建立信任闭环。Agent 完成意图解析后，不是返回纯文本，而是动态渲染对应的 **GUI 交互卡片**——数据卡片带趋势条形图、表格卡片带状态标签、表单卡片带完整字段、审批卡片带操作按钮、工作流卡片带时间线节点。每张卡片都是可交互的业务界面组件，用户可以在对话流中直接完成操作。

这一设计与 Salesforce Agentforce Experience Layer 的核心思想高度一致：将"Agent 做什么"与"如何呈现"分离，Agent 负责交付数据和行为意图，前端负责将其渲染为最合适的交互形态。

对话界面支持六种动态 GUI 卡片类型，覆盖企业业务的核心交互场景：

| 卡片类型 | 触发场景 | 核心交互能力 |
|---------|----------|-------------|
| **Data Card** | "查看华东区Q3销售业绩" | 大数字指标 + 趋势条形图 + 操作按钮 |
| **Table Card** | "显示本月销售Top 10" | 可排序表格 + 状态徽章 + 排名标识 |
| **Form Card** | "创建客户跟进任务" | 多字段表单（文本/选择/日期/文本域） |
| **Chart Card** | "查看当前季度销售漏斗" | 漏斗条形图 + 转化率标注 + 总额指标 |
| **Approval Card** | "审批张明提交的报销" | 信息摘要网格 + 同意/驳回/转交按钮 |
| **Workflow Card** | "启动新员工入职流程" | 步骤时间线 + 状态指示 + 催办操作 |

这些卡片的渲染基于 **Declarative Generative UI** 模式——Agent 返回结构化 UI 规范（JSON Schema），前端在组件约束和样式体系内完成渲染，兼顾安全性与灵活性。这与 Google A2UI 协议、OpenJSON-UI 规范的技术方向一致。

### 2.2 智能驾驶舱 — Agentic GUI 的业务聚合呈现

智能驾驶舱是 YonClaw 最具创新性的模块，它代表了从"传统固定 GUI"向"Agent 动态生成 GUI"的根本性转变。

#### 2.2.1 设计哲学：不是页面，是界面表面

传统的企业驾驶舱（Dashboard）是预先设计好的固定页面：开发者在代码中硬编码每个图表的位置、数据源和样式，用户只能被动消费。YonClaw 的智能驾驶舱完全不同——它是一个**临时组装、按需生成、多 Agent 协作渲染**的动态界面表面。

当用户通过对话表达"帮我创建一个销售业绩驾驶舱"的意图时，YonClaw 主智能体执行以下编排逻辑：

1. **意图解析**：识别用户需要"销售数据聚合展示"
2. **Agent 调度**：同时唤醒"销售助手"和"营销智脑"两个子智能体
3. **数据聚合**：销售助手提供业绩数据、客户排行、漏斗转化；营销智脑提供渠道ROI、获客成本、投放数据
4. **UI 组装**：主智能体根据数据特征自动选择最合适的 widget 类型——指标卡片用于 KPI、条形图用于趋势、表格用于排行、看板用于漏斗、时间线用于进度
5. **实时渲染**：所有 widget 按网格布局动态渲染，形成完整的驾驶舱界面

这与 Salesforce Headless 360 提出的 "Agentforce Experience Layer" 概念高度对应——Agent 不是返回数据让前端自己决定如何展示，而是直接交付"体验"本身。

#### 2.2.2 多智能体协作与运行时日志

每个智能驾驶舱都关联一个**主智能体**和多个**协作智能体**。以"销售业绩驾驶舱"为例：

- **主智能体**：销售助手（负责整体数据聚合和界面组装）
- **协作智能体**：营销智脑（提供渠道转化数据）、财务管家（提供预算执行数据）

驾驶舱头部显示所有关联 Agent 的头像，用户可点击展开查看每个 Agent 的实时状态（运行中/空闲/异常）、调用次数、技能列表。这种设计让用户清楚了解"谁在为我工作"。

右侧的**执行日志面板**是另一个关键创新。它实时记录多 Agent 协作的完整执行过程——时间戳、Agent 头像、执行动作、执行结果（成功/警告/错误）、详细说明。这不仅提供了过程透明性，更重要的是当业务出现问题时，用户可以通过日志追溯是哪个 Agent、哪一步操作导致了异常，实现**可观测的 Agent 编排**。

#### 2.2.3 主智能体会话入口

每个驾驶舱都提供一个"与 XX助手 会话"的入口按钮。这意味着用户不需要离开驾驶舱就可以与主智能体对话——可以在查看数据的同时，直接提问"为什么华东区业绩下滑"，主智能体会基于当前驾驶舱的上下文给出针对性分析，并可能动态更新驾驶舱中的图表和指标。

这种"界面即对话、对话即界面"的双向融合，是 Agentic UI 的终极形态。

### 2.3 智能团队 — 双维度智能体管理

智能团队模块是企业所有业务智能体的统一管理中心。每个智能体提供**管理视图**和**业务视图**两个维度，分别服务不同角色的需求。

#### 2.3.1 管理视图：运维人员的监控中心

管理视角关注智能体本身的运行状态，包括：

- **基础信息**：智能体介绍、技能列表、维护者、运行状态（运行中/空闲/异常/构建中）
- **调用趋势**：近7天的调用量柱状图和成功率曲线
- **核心指标**：总调用次数、服务用户数、平均响应时间、成功率
- **最近对话**：该智能体最近处理的5条用户请求记录

这一维度类似传统系统的"服务监控"，但监控对象从"微服务"变成了"智能体"——每个 Agent 都是一个可被独立观测、启停、配置的智能服务单元。

#### 2.3.2 业务视图：业务人员的工作台

业务视角关注智能体当前处理和分析的业务内容，是完全不同的信息呈现：

- **KPI 卡片**：4个核心业务指标（如销售额、新增客户、成交率、客单价），每个指标附带同比变化和趋势方向
- **业务图表**：2张动态图表（如月度业绩趋势、销售漏斗转化），数据由 Agent 实时从业务系统抓取
- **业务表格**：Top 客户排行、待办事项列表等结构化数据
- **业务洞察**：Agent 基于数据分析主动生成的预警和建议（如"3个重点客户本周无跟进记录，存在流失风险"）

业务视图的设计逻辑是：用户不需要去 CRM 查客户、去 BI 看报表、去邮件看预警——所有这些信息由 Agent 自动聚合，以最适合阅读的卡片形态呈现。这正是 SAP Joule 所追求的" woven-in"体验——AI 不是外挂工具，而是深度融入业务操作的智能层。

### 2.4 智能工作流 — 多 Agent 可视化编排

智能工作流模块提供**可视化流程编排器**，让用户可以通过拖拽节点的方式定义多智能体协作的业务流程。这是从"单 Agent 对话"到"多 Agent 自动化协作"的关键跃迁。

#### 2.4.1 节点类型设计

工作流支持7种节点类型，覆盖业务流程编排的完整需求：

| 节点类型 | 功能 | 颜色标识 |
|---------|------|---------|
| **触发器** | 定义流程启动方式（手动/定时/事件） | 琥珀色 |
| **智能体** | 调用指定子智能体执行任务 | 靛蓝色 |
| **条件** | 基于数据判断进行分支路由 | 粉色 |
| **审批** | 人工审批节点（需指定审批人） | 紫色 |
| **转换** | 数据格式转换或清洗 | 翠绿色 |
| **通知** | 发送消息（邮件/App/短信） | 蓝色 |
| **结束** | 流程终止节点 | 灰色 |

以"新员工入职流程"为例：触发器（新员工信息提交）→ HR助手（创建档案）→ IT助手（开通账号）→ 审批（直属经理确认）→ 条件（是否通过）→ 行政（工位分配）→ IT助手（权限开通）→ HR助手（培训安排）→ 通知（发送欢迎邮件）→ 结束。

#### 2.4.2 可视化编排体验

编排器采用画布式设计：
- 左侧工具栏提供可拖拽的节点类型
- 中央画布显示节点卡片和 SVG 贝塞尔曲线连接
- 右侧属性面板显示选中节点的详细配置（关联智能体、触发事件、Cron表达式、判断条件、审批人、通知渠道等）
- 节点支持状态着色（空闲/执行中/已完成/异常），便于运行时可视化追踪

这与微软 Azure AI Agent Service 提出的多 Agent 编排架构（Supervisor Pattern / Adaptive Network）形成产品层对应——底层是 Agent 之间的任务传递协议（A2A），上层是用户可以直观理解和维护的可视化流程。

### 2.5 友智库 — 企业知识资产的智能管理

友智库是企业知识资产的集中管理与智能检索模块，承担"组织记忆"的职责。在 Agentic 时代，知识库不仅是给人看的文档，更是给 Agent 用的"记忆"——Agent 在处理业务时需要实时检索相关知识来提供准确、合规的回答。

#### 2.5.1 知识分类体系

友智库采用七级分类架构：

| 分类 | 定位 | 内容示例 |
|------|------|---------|
| **产品文档** | 产品功能与能力说明 | 白皮书、功能清单、更新日志 |
| **解决方案** | 行业/场景方法论 | 数字化转型指南、实施方法论 |
| **制度规范** | 企业治理文件 | 数据安全制度、API开发规范 |
| **客户案例** | 标杆实践沉淀 | 华为财务共享、美的智能制造 |
| **操作指南** | 使用手册与教程 | 平台操作手册、配置最佳实践 |
| **常见问题** | FAQ 与问答沉淀 | 触发器配置、权限问题排查 |

每篇文档包含标题、摘要、分类、标签、作者、阅读量、点赞数等多维元数据，支持全文检索和分类筛选。

#### 2.5.2 与智能体的协同

友智库不是独立模块——它与智能团队深度联动：

- 当用户向销售助手提问时，Agent 会自动检索友智库中的"产品文档"和"客户案例"来丰富回答
- 当法务合规官审查合同时，Agent 会检索"制度规范"中的相关条款进行合规比对
- 当创建新的智能体时，开发者可以从友智库中引用已有文档作为知识基础

这种"知识即服务（Knowledge-as-a-Service）"的设计理念，确保 Agent 的回答不仅基于模型训练数据，更基于企业自身的真实知识资产，实现**企业专属的智能**。

### 2.6 左侧导航架构

左侧边栏是整个平台的导航中枢，采用 Slack 式深色侧边栏设计，所有模块均支持点击展开/折叠，历史记录支持快速回溯。

```
YonClaw Logo              → 返回首页
新建任务 ⌘K               → 快速创建新对话
─────────────────────────
智能团队                  → 智能体列表页
智能驾驶舱                → 驾驶舱列表页
智能工作流                → 工作流列表页
友智库                    → 知识文库页
定时任务                  → 定时任务管理
技能中心                  → 技能市场与管理
─────────────────────────
我的智能体 (8)  [可展开]   → 8个业务Agent快捷入口
驾驶舱 (5)      [可展开]   → 5个驾驶舱快捷入口
工作流 (4)      [可展开]   → 4个工作流快捷入口
历史对话        [可展开]   → 10条最近对话记录
─────────────────────────

```

这一导航设计的核心原则是：**高频入口平铺可见，低频入口折叠收纳，所有内容均可点击到达详情**。用户的任何历史会话、任何工作空间、任何智能体，都可以从侧边栏一键直达。

## 三、设计原则与技术架构

### 3.1 五大核心设计原则

**原则一：对话即入口（Conversation as Entry）**

用户不需要记住功能在哪里、菜单怎么点——任何业务需求都可以通过自然语言表达。YonClaw 主智能体负责将自然语言转化为结构化任务，再调度对应的子智能体执行。这与 Salesforce 提出的 "Everything is an API, MCP tool, or CLI command and agents can use all of it" 的理念一致。

**原则二：界面按需生成（UI on Demand）**

界面不是预先设计好的，而是根据对话上下文、业务数据和用户角色**动态组装**的。同一个"查询销售数据"的意图，高管看到驾驶舱级别的聚合视图，销售经理看到团队级别的明细表格，销售人员看到个人任务列表。这是真正的 **Context-Aware Generative UI**。

**原则三：多 Agent 透明协作（Transparent Multi-Agent Orchestration）**

用户需要知道"谁在为我工作"以及"工作进展如何"。智能驾驶舱的 Agent 头像、执行日志、状态指示，都是透明协作的体现。这与 SAP Joule 的多 Agent 编排设计（Agent 之间可以相互调用）形成产品层呼应。

**原则四：渐进式交互（Progressive Disclosure）**

信息呈现遵循渐进原则：对话界面提供即时反馈 → 动态卡片提供可操作界面 → 驾驶舱提供聚合视图 → 工作流提供自动化编排。用户可以根据需求深度，在不同层级间自由切换，不会感到信息过载。

**原则五：企业级信任（Enterprise Trust）**

所有 AI 生成的内容必须可追溯、可审计、可人工干预。审批卡片中的"Human-in-the-Loop"确认节点、工作流中的审批节点、业务洞察中的来源归因，都是信任设计的一部分。这与 SAP Joule 遵循企业级角色权限模型的设计理念一致。

### 3.2 关键技术协议映射

| YonClaw 模块 | 对应技术协议/标准 | 作用 |
|-------------|-----------------|------|
| Agent 与工具/数据连接 | **MCP** (Model Context Protocol) | 统一 Agent 与外部系统（CRM/ERP/HCM）的集成标准 |
| Agent 生成 UI 描述 | **A2UI** / OpenJSON-UI | Agent 返回结构化 UI 规范，前端跨平台渲染 |
| Agent 间任务传递 | **A2A** (Agent-to-Agent) | 标准化多 Agent 之间的任务委托和协作 |
| 工作流节点编排 | **AG-UI** 事件流 | 支持 UI 增量更新和流式渲染 |
| 知识检索 | **RAG** + Knowledge Graph | Agent 从企业知识库实时检索增强回答 |

## 四、原型交互说明

### 4.1 典型用户旅程

**场景一：销售总监查看业绩**

1. 用户在首页输入"查看华东区Q3销售业绩"
2. YonClaw 主智能体唤醒"销售助手"Agent
3. Agent 流式回复文字分析，随后动态渲染 Data Card（销售额 ¥2,847万 +23%）和趋势条形图
4. 用户点击卡片上的"查看详情"按钮，跳转至"销售业绩驾驶舱"
5. 驾驶舱中销售助手和营销智脑联合渲染——左侧是业绩趋势图和Top客户表，右侧是执行日志（两个Agent的协作过程实时可见）
6. 用户点击"与销售助手会话"，直接在驾驶舱内发起对话追问"为什么9月业绩下降"

**场景二：HR处理新员工入职**

1. 用户在首页输入"启动新员工王小明的入职流程"
2. 主智能体触发"新员工入职"工作流
3. 工作流自动执行：HR助手创建档案 → IT助手开通账号 → 通知经理审批 → 行政分配工位
4. 用户进入"入职管理台"驾驶舱，查看时间线进度（信息录入✓ / IT开通✓ / 经理确认→进行中）
5. 右侧日志实时显示每个Agent的执行记录
6. 经理审批通过后，工作流自动继续后续步骤

**场景三：查找产品资料**

1. 用户点击左侧"友智库"进入文库
2. 在搜索框输入"智能财务"
3. 友智库返回匹配的文档列表（YonBIP智能财务白皮书、智能审批配置指南等）
4. 用户点击白皮书，查看完整内容，同时可以看到"此文档已被销售助手引用 47 次"

### 4.2 视觉设计规范

- **色彩体系**：深色主题为主（#0f0f1a 背景），品牌色为红橙渐变（#ef4444 → #f97316），成功态翠绿、警告态琥珀、错误态红色
- **卡片设计**：圆角 12px、微边框（white/4%）、hover 时边框提亮 + 背景微增，所有卡片统一动画（fade-in + slide-up 500ms）
- **字体层级**：标题 18px semibold、正文 14px regular、辅助 12px / 11px、元数据 10px
- **间距体系**：遵循 4px 网格，模块间距 24px，卡片内边距 16-20px
- **图标体系**：统一使用 Lucide 图标库，尺寸 16/20/24px 三档

## 五、演进路线与未来展望

### 5.1 当前版本能力（v3.0）

- 自然语言对话 + 6种动态 GUI 卡片
- 8个业务智能体（管理+业务双维度）
- 5个智能驾驶舱（多Agent协作 + 实时日志）
- 5个预设工作流（可视化编排器）
- 15篇文档的友智库（分类 + 搜索 + 详情）

### 5.2 近期演进方向

- **语音交互**：支持语音输入和 TTS 播报，实现真正的"动口不动手"
- **跨平台自适应**：同一驾驶舱在 Web、移动端、企业微信/钉钉中自适应渲染
- **工作流触发器增强**：支持更多事件源（邮件收到、数据变更、定时 Cron）
- **友智库 AI 摘要**：Agent 自动为长文档生成智能摘要和问答对

### 5.3 远期愿景

YonClaw 的终极形态是一个**完全 Headless 的企业智能中台**——所有业务能力以 API/MCP 形式暴露给 Agent，所有界面由 Agent 根据用户意图和上下文动态生成，用户通过自然语言与单一入口交互即可驱动整个企业的数字化运营。这不是一个产品，而是一种全新的企业软件交互范式。
